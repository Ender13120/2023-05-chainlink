Hi all, answering some questions that get repeated in DMs quite often
  • for the purposes of this audit, assume n = 3f + 1. Let's say f = 5 => n = 16. Since we require (n+f)/2 + 1 signatures, this means (16+5)/2+1 = 11 signatures.
  • The rate limiting in ramps if different from the token pool rate limiting in the sense that ramps use USD and pools use the token itself. This is intentional, as the ramps rate limit over many tokens and require a shared denomination. The pools only rate limit a single token.
  • If s_wrappedNative is set to address(0), this means there is no auto wrapping native, therefore, native payments are disabled.
  • For LINK NOP payments in onRamp, some entity needs to supply the LINK tokens so the non-link tokens can be redeemed. This can be the owner but technically anyone is able to send link to the contract to unlock the non-link fee tokens.

Can I ask how the upgrade process for on-ramp and off-ramp works?
Rens | Chainlink 06/09/2023 07:51
We always upgrade the entire lane, so onRamp, commitStore and offRamp. We deploy the new lane and allow the new lane on the chain contracts (priceRegistry, token pools and destination router). When ready to upgrade we point the source router to the new onRamp and from then on all txs will go through the new lane. The old onRamp is immediately no longer used, but the commitStore and offRamp can still process the txs that are in flight for the old lane.
